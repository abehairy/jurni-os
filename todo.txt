## Shell / UX — done this session
- Removed Timeline, People, Patterns as separate pages. They're group modes
  inside the Landscape now. Landscape IS the app.
- CSS-variable theme system with warm-light (default) / warm-dark toggle.
  Adding a third theme is just another `[data-theme="x"]` block in index.css.
- Landscape goes edge-to-edge (no more "card on page" look). Sidebar + main
  share one surface.
- Pinning: pin button in the drill drawer, live pinned list in sidebar.
  Click a pin → switches group mode if needed, opens the drawer instantly.
  Backed by a `pins` table with `UNIQUE(tile_key, group_mode)`.

## Entity resolution / identity graph — deferred, build later
Problem: "Amr" collides across 2+ people. Gets worse when calendar/email adds
bare names with no context. See conversation for full audit.

Decided shape (fluid, not rigid):
- mentions table (every extracted name + context)
- identity_links table (must_link / cannot_link constraints from user clicks)
- identity = clustering computed over mentions (union-find + context overlap)
- strong keys (emails, phones) = privileged edges in same graph
- UX: in-place split/merge on a tile ("this isn't the same person")
- No profiles table, no "review center" screen, no onboarding addition

Phases:
1. Plumbing (mentions + identity_links tables, non-breaking)
2. Strong-key ingestion (once contacts/calendar connectors exist)
3. LLM disambiguation in thread pass
4. In-place split/merge affordance on People tiles

Known leak to fix NOW regardless: user's email (e.g. "ahmed@behairy.me") is
being extracted as a person entity. identity filter only matches name strings,
not emails. Need to add the user's email as an alias, OR teach extraction not
to extract emails as people at all.
